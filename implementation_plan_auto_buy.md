# Implementation Plan: Bitkub Transaction History Auto-Sync

เพื่อแก้ปัญหาการที่ผู้ใช้ต้องมากดบันทึกประวัติการซื้อ BTC (DCA) ด้วยมือ เราจะพัฒนาระบบดึงประวัติการซื้อขายจริงย้อนหลังจาก Bitkub API อัตโนมัติ โดยแบ่งงานออกเป็น 4 เฟสหลัก ดังนี้

---

## User Review Required

> [!IMPORTANT]
> **1. การเก็บคีย์ API และความปลอดภัย:** 
> ปัจจุบันแอปพลิเคชันเก็บ Bitkub API Key/Secret ไว้ที่ `.env.local` บนฝั่งเซิร์ฟเวอร์ ซึ่งแปลว่ารองรับบัญชี Bitkub ของผู้ดูแลระบบ (Admin) เพียงบัญชีเดียวเป็นหลัก หากต้องการให้ผู้ใช้อื่นๆ สามารถล็อกอินและใส่คีย์ของตัวเองได้ในอนาคต จะต้องย้ายไปเข้ารหัสและเก็บคีย์ใน Firestore (ตามตาราง `exchange_accounts` ในดีไซน์ดั้งเดิม)
>
> ในแผนนี้ จะยึดการใช้ API Key จาก `.env.local` เพื่อให้บริการผู้ใช้งาน Admin เป็นหลักก่อน เพื่อให้ทำงานได้ง่ายและตรงกับสภาพระบบปัจจุบันที่สุด

> [!WARNING]
> **2. ข้อจำกัดข้อมูลย้อนหลัง 90 วัน ของ Bitkub:**
> API ของ Bitkub จะส่งข้อมูลประวัติคำสั่งซื้อ (`my-order-history`) ย้อนหลังได้ไม่เกิน 90 วัน หากผู้ใช้ทำการซื้อก่อนหน้านั้นและต้องการเก็บไว้ในระบบ จะต้องใช้วิธีกรอกประวัติเพิ่มเติมด้วยมือ (AddTransaction) ร่วมด้วย

## Proposed Changes

การแบ่งงานออกเป็น 4 เฟสหลัก:

### Phase 1: พัฒนา Backend API Route สำหรับดึงประวัติเทรด
สร้าง API endpoint ใหม่เพื่อคุยกับ Bitkub API V3 ป้องกันปัญหา CORS และซ่อนความลับของ API Signatures ไว้หลังบ้าน

#### [NEW] [route.ts](file:///h:/06-Projects/bitcoin-dca-tracker/app/api/bitkub-history/route.ts)
* สร้าง Route `GET /api/bitkub-history`
* ส่งคำขอไปยัง `GET https://api.bitkub.com/api/v3/market/my-order-history?sym=THB_BTC`
* คำนวณลายเซ็น HMAC-SHA256 โดยใช้คีย์และทรานแซกชัน Timestamp จากระบบเซิร์ฟเวอร์
* กรองเฉพาะรายการที่เป็นการซื้อ (`side === 'buy'`) และจับคู่อินพุตให้อยู่ในฟอร์แมตที่ระบบใช้งานได้:
  * `orderId` (สำหรับเช็กข้อมูลซ้ำ)
  * `amountTHB` (จำนวนเงินบาทที่จ่ายไป)
  * `btcAmount` (จำนวน BTC ที่ได้รับจริง)
  * `btcPriceAtBuy` (ราคาต่อ 1 BTC ณ จุดที่ซื้อ)
  * `date` (เวลาที่ระบบบันทึกรายการคำสั่งซื้อสำเร็จ)

---

### Phase 2: ระบบซิงก์ข้อมูลและป้องกันการเขียนซ้ำ (Sync & Deduplication)
เขียนฟังก์ชันฝั่ง Client-side เพื่อรับข้อมูลจาก API และเขียนลงฐานข้อมูล Firestore ของผู้ใช้งาน

#### [MODIFY] [page.tsx](file:///h:/06-Projects/bitcoin-dca-tracker/app/page.tsx)
* เพิ่มฟังก์ชัน `syncBitkubHistory()` ที่จะส่งคำขอไปที่ `/api/bitkub-history`
* ดึงข้อมูลธุรกรรมที่จัดเก็บอยู่แล้วใน Firestore เพื่อนำมาเปรียบเทียบเชิงอ้างอิง
* ทำการกรอง (Filter) ตรวจสอบว่า `orderId` ของ Bitkub API นั้น มีอยู่ในคอลเลกชัน `transactions` ของ Firestore แล้วหรือยัง เพื่อป้องกันการบันทึกรายการซ้ำ
* หากเป็นรายการใหม่ ให้เพิ่มเข้าไปใน Firestore แบบไร้รอยต่อ

---

สร้างปุ่มกดใช้งานง่ายบน Dashboard เพื่อกระตุ้นการซิงก์ข้อมูล และเพิ่มตัวบ่งชี้ว่าธุรกรรมใดมาจาก Bitkub

ทำให้ผู้ใช้แทบไม่ต้องกดปุ่มเอง โดยให้ระบบซิงก์ประวัติทันทีเมื่อเปิดแอปพลิเคชันและล็อกอิน

#### [MODIFY] [page.tsx](file:///h:/06-Projects/bitcoin-dca-tracker/app/page.tsx)

## Verification Plan

### Automated Tests
* ไม่มีชุดคำสั่งเทสอัตโนมัติ (เนื่องจากต้องการการจำลองการตอบสนองของ Exchange) แต่จะรันคำสั่งตรวจสอบ Lint และ Type Checking:
  ```bash
  npm run lint
  ```

### Manual Verification
1. **การตรวจสอบ API Route:** เรียกไปที่ `/api/bitkub-history` ผ่านบราวเซอร์/Postman เพื่อเช็กว่าสามารถดึงรายการคำสั่งซื้อจากบัญชีออกมาได้ถูกต้องและอยู่ในฟอร์แมต JSON ที่กำหนด
2. **การซิงก์ข้อมูลซ้ำ:** ลองกดปุ่มซิงก์มากกว่า 1 ครั้ง เพื่อดูว่ายอดเงินสะสมและจำนวนรายการในฐานข้อมูล Firestore เพิ่มขึ้นผิดปกติหรือไม่ (ต้องคงเดิมถ้าไม่มีรายการใหม่)
3. **การเข้าสู่ระบบแล้วซิงก์ทันที:** ล็อกเอ้าต์แล้วล็อกอินใหม่ สังเกตว่าระบบมีเอฟเฟกต์การซิงก์ข้อมูลอัตโนมัติทำงานหรือไม่
